# Libral Core ベース構想マップ

**「ベースとなるコアなので、実際に実装しません」** — このコアを土台に、以下のようなアイデアを形にしていくための対応表です。

---

## コアが提供する土台（おさらい）

| レイヤー | 主な機能 |
|---------|----------|
| **通信** | Telegram Webhook, メール/Webhook アダプタ, トランスポート（Aegis 送信） |
| **認証・暗号** | Aegis-PGP（暗号・署名・検証・WKD）, スタンプ作成時の通知 |
| **AI・KB** | AI Bridge, AI Router, Evaluator, OSS Manager, Embedding, **KB System（独立）** |
| **運用** | モジュールレジストリ, ストレージ（ユーザー/イベント/スタンプ/メトリクス）, Redis, WebSocket, イベントバス |
| **UI** | C3 Console（Monitor / Control / Creation）, KB Editor, ダッシュボード, GPG 設定, 決済デモ など |

---

## 構想一覧とコアとの対応

### 1. Libral.秘話（ミニアプリ・モジュール型プラットフォーム）

| 項目 | 内容 |
|------|------|
| **概要** | 「ほしい機能を、押すだけ。」— 機能を選んで組み立てるモジュール式ミニアプリ。Telegram を前面に出さない汎用表現。 |
| **コアで使うもの** | モジュールレジストリ, 通信ゲートウェイ（Telegram/Webhook/Email）, イベント, 設定/認証。C3 Console を「自分専用アプリ」の操作UIとして拡張。 |
| **新規に作るもの** | ロゴ・タグライン, 「押すだけ」のボタンUI, モジュールのオン/オフ・並び順の管理。 |

| 項目 | 内容 |
|------|------|
| **概要** | 全17モジュールの完全仕様書（Core / AI / Operation / Functional 層）。`modules_full.yaml` + 人間向け説明。 |

### 3. MultiPost（匿名代理投稿 Bot）

| 項目 | 内容 |
|------|------|
| **概要** | Telegram で匿名にグループ・チャンネルへ代理投稿。投稿者IDは出さず、Bot 名で投稿。 |
| **コアで使うもの** | **通信（Telegram アダプタ）**, イベントログ, ストレージ（下書き・ログ・BAN 一覧は別スキーマで拡張可能）。Aegis で「匿名ID」をハッシュ化するような拡張も可能。 |
| **新規に作るもの** | 投稿フロー（ステップUI）, 投稿先設定・ログ・BAN 管理, `settings.json` / `pending_posts.json` 等の Bot 専用ストア。 |

### 5. パーソナル・ハイブリッド VPN チェーン

| 項目 | 内容 |
|------|------|
| **概要** | 個人・近親者向け。マルチホップ VPN（WireGuard 等）, Hetzner + Cloudflare, 各端末から簡単接続。 |
| **コアで使うもの** | コアは「アプリ基盤」なので直接は使わない。設計書・運用ポリシーは **ドキュメントとして** 同じリポジトリの `docs/` や別リポで管理すると、認証・監査（Aegis/Governance）と方針を揃えやすい。 |

## まとめ

- **コア** = 通信・認証・AI/KB・決済・イベント・スタンプ・モジュール管理・C3 Console までを「共通基盤」として提供。
- **各構想** = その上に「匿名投稿」「モデレーター」「スタンプ Bot」「SNS」「VPN 設計書」「KB システム」などを**アプリ or ドキュメント**として乗せるイメージ。
- 実装は各リポジトリ・各ドキュメントで行い、コアは**実装しない・変えすぎない**ことで、どのアイデアでも同じ土台を前提にしやすくする。

必要に応じて、上表を「実装状況」「優先度」「リポジトリリンク」で更新していくと、全体の地図として使いやすくなります。
